## 3. 核心架构：PEV Agent 框架

PEV (Planner-Executor-Verifier) 是系统的核心 RAG 框架，支持迭代式问答：

```
用户提问
    │
    ▼
┌─────────────────────────────────────────┐
│               PLANNER                    │
│  LLM 将问题分解为子任务列表               │
│  例: "什么是二叉树" →                     │
│    [1] 检索二叉树定义                     │
│    [2] 检索二叉树性质                     │
│    [3] 检索二叉树应用                     │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│              EXECUTOR                    │
│  For each subtask:                       │
│  ├─ FAISS 向量检索                       │
│  ├─ BM25 稀疏检索                        │
│  ├─ 知识图谱检索 (可选)                   │
│  ├─ 合并去重检索结果 (按 chunk_id)        │
│  └─ LLM 基于检索结果合成答案 + 引用编号   │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│              VERIFIER                    │
│  LLM 验证 (4 维度):                      │
│  ├─ 充分性 (sufficiency)                 │
│  ├─ 准确性 (accuracy)                    │
│  ├─ 引用正确性 (citation correctness)     │
│  └─ 一致性 (consistency)                 │
│                                          │
│  如果不充分 → 补充任务 → 回到 EXECUTOR    │
│  最多迭代 3 轮                           │
└────────────────┬────────────────────────┘
                 │
                 ▼
            最终答案 + 引用列表
```

## 4. 智能推荐控制器

`controller.py` 中的推荐控制器在回答前进行智能决策：

```
用户提问
    │
    ▼
┌─────────────────────────────┐
│    LLM 判断 (controller)     │
│  ├─ 是否需要推荐前置知识？    │  → 若是: pre_knowledge_search()
│  └─ 是否需要推荐练习题？      │  → 若是: qa_retriever.search()
└──────────────┬──────────────┘
               │
               ▼
         PEV Agent 生成主答案
               │
               ▼
    返回: 答案 + 决策理由 + 前置知识 + 推荐题目
```

### 配置结构 (`config/example.yaml`)

```yaml
llm:
  use: 'dmxapi'                    # 当前使用的 Provider
  dmxapi:
    api_key: '...'
    api_base: 'https://...'
    model: 'gpt-4o-mini'
    temperature: 0.3
    top_p: 0.9
    max_tokens: 4096

embedding:
  use: 'siliconflow'               # 当前使用的 Embedding Provider
  siliconflow:
    api_key: '...'
    model: 'BAAI/bge-large-zh-v1.5'
```

## 7. 检索系统

### 三路检索融合

```
查询文本
    │
    ├─── FAISS 向量检索 ──── 语义相似度 Top-K
    │    (faiss2.index, SiliconFlow embedding)
    │
    ├─── BM25 稀疏检索 ──── 关键词匹配 Top-K
    │    (jieba 分词, BM25Okapi)
    │
    └─── 前置知识检索 ───── 实体匹配 + 知识图谱
         (pre_knowledge_graph.json, ~45 算法概念)
         │
         ▼
    合并去重 → 最终检索结果
```

### 题库检索 (QA Retrieval)

```
查询文本
    │
    ├─── TF-IDF 相似度计算
    │
    ├─── 关键词匹配
    │
    └─── 混合评分排序
         │
         ▼
    匹配的练习题列表
```

| 类别 | 用途 |

## 10. 备课系统 (Preparation)

### 四阶段流水线

```
Stage 1: PARSE（解析）
  输入: PPT/PDF/Markdown 文件
  处理: python-pptx / MinerU / 文本切分
  输出: 结构化大纲 + 内容块

Stage 2: DECIDE（决策）
  输入: 结构化大纲
  处理: LLM 识别思政融合点
  输出: 融合点列表 + 插入位置

Stage 3: RETRIEVE（检索）
  输入: 融合点关键词
  处理: 关键词提取 → 思政内容检索 → 语义重排
  输出: 匹配的思政素材

Stage 4: OPTIMIZE（优化）
  输入: 原始内容 + 思政素材
  处理: LLM 内容生成 → 风格适配 → 页面优化
  输出: Marp 幻灯片 / 优化后的 PPT
```

## 11. Embedding Pipeline (`embedder.py`)

数据准备脚本（非运行时模块）：
1. 读取 Markdown 教材文件
2. 按标题结构切分（章/节层次）
3. 精细切分到最大 300 字符
4. 通过 LLM 提取元数据
5. 通过 SiliconFlow API 批量 Embedding
6. 构建 FAISS IndexFlatL2 并保存到磁盘

## 14. 当前局限

| 项 | 说明 |
|----|------|
| 题库硬编码 | `problem_bank.py` 仅少量题目，需接入 backend 数据 |
| 代码执行实验性 | `code_runner.py` 调用 judgend 服务，集成度待提升 |
| Embedding 需预构建 | FAISS 索引需离线构建，无在线增量更新 |
| 知识图谱静态 | 知识图谱数据为 JSON 文件，无在线编辑能力 |
| 会话无鉴权 | API 直接传 user_id，未校验 JWT |
| 备课系统实验性 | 思政融合逻辑依赖 LLM 质量，输出稳定性待验证 |
