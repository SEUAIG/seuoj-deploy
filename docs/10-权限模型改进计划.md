# SEUOJ 权限模型改进计划

## Context

当前系统的权限控制存在以下结构性问题：

1. **权限检查散落**：ProblemAccessService、ClassService.assertCanManageClass()、ProblemSetService.assertCanManageProblemSet() 各自独立实现，模式不统一
2. **角色命名不当**：USER 应为 STUDENT 以匹配教学场景语义；SUPER_ADMIN 与 ADMIN 缺少明确的职责区分
3. **资源归属不一致**：problem 无 creator 字段、problem_set 用 `owner_user_id`、class_info 用 `teacher_user_id`、contest 靠 `contest_manager_rel`
4. **多张专用关联表**：`problem_set_invited_member_rel`、`contest_manager_rel`、`user_problem_set_rel` 各自表达"某用户对某资源有权限"，但 schema 各异
5. **班级关联题单无生命周期**：`class_problem_set_rel` 只是裸链接，教师无法准备"未来作业"

本计划基于用户草案，对权限模型做破坏性重构，以架构简洁和扩展性为第一优先级。

---

## 一、角色重整：USER → STUDENT，保留四角色

### 目标角色

| 角色 | code | 授权方式 | 默认权限 |
|------|------|---------|---------|
| 学生 | STUDENT | 注册自动获得 | 无资源默认权限，仅可访问公开资源和被授权资源 |
| 教师 | TEACHER | ADMIN/SUPER_ADMIN 通过面板授权 | 可读所有资源 + 可创建新资源 |
| 管理员 | ADMIN | SUPER_ADMIN 通过面板授权 | 读写所有资源 |
| 超级管理员 | SUPER_ADMIN | 仅数据库直接操作 | 读写所有资源 + **可授予/撤销 ADMIN 角色**（唯一区别） |

### SUPER_ADMIN 与 ADMIN 的唯一区别

SUPER_ADMIN 可以通过管理面板将用户设为 ADMIN 或撤销 ADMIN 角色；ADMIN 只能操作 TEACHER/STUDENT 角色。除此之外二者权限完全相同。

### 变更

- **重命名** `USER` → `STUDENT`（code 值变更）
- **RoleType.java** 改为 `STUDENT`, `TEACHER`, `ADMIN`, `SUPER_ADMIN` 四个枚举值
- 删除 `hasAdminRole(Collection)` 等静态方法（逻辑移入 PermissionService）
- **user_role 种子数据**：4 行（id=1 STUDENT, id=2 TEACHER, id=3 ADMIN, id=4 SUPER_ADMIN）
- **注册流程**：`AuthService.register()` 默认分配 STUDENT 角色（替代当前的 USER）
- **登录响应**：`LoginVO.role` 返回 `"student"` / `"teacher"` / `"admin"` / `"superadmin"`

## 二、统一资源权限表：resource_permission

### 设计原则

用一张表取代所有"某用户对某资源有额外权限"的专用关联表。

### Schema

```sql
CREATE TABLE resource_permission (
    id                BIGINT       NOT NULL AUTO_INCREMENT,
    resource_type     VARCHAR(32)  NOT NULL COMMENT 'PROBLEM|PROBLEM_SET|CLASS|CONTEST',
    resource_id       BIGINT       NOT NULL,
    user_id           BIGINT       NOT NULL,
    permission        VARCHAR(16)  NOT NULL COMMENT 'READ|WRITE',
    granted_by        BIGINT       NULL,
    created_at        TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    is_del            TINYINT(1)   NOT NULL DEFAULT 0,
    active_key        VARCHAR(192) GENERATED ALWAYS AS (
        CASE WHEN is_del = 0
             THEN CONCAT(resource_type, '#', resource_id, '#', user_id, '#', permission)
             ELSE NULL END
    ) STORED NULL,
    PRIMARY KEY (id),
    UNIQUE INDEX uk_active (active_key),
    INDEX idx_user (user_id, resource_type),
    INDEX idx_resource (resource_type, resource_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 替代关系

| 被替代的表 | 在 resource_permission 中的表达 |
|-----------|-------------------------------|
| `problem_set_invited_member_rel` | `(PROBLEM_SET, ps_id, user_id, READ/WRITE)` |
| `contest_manager_rel` | `(CONTEST, contest_id, user_id, WRITE)` |
| `user_problem_set_rel` | `(PROBLEM_SET, ps_id, user_id, READ)` |

### 授权规则

- **WRITE 蕴含 READ**：拥有 WRITE 权限的用户自动拥有 READ
- **授权传播**：拥有 WRITE 权限的用户可以对同一资源授予他人任意权限
- **创建者自动授权**：资源创建时，在 `resource_permission` 中自动插入一条 `WRITE` 记录给创建者。权限判定**不**通过 `created_by_user_id` 字段进行，该字段仅作为信息记录（谁创建的），所有权限检查统一走 `resource_permission` 表

## 三、资源实体归属统一

### 设计原则

`created_by_user_id` 仅作为**信息字段**（记录谁创建了这个资源，用于审计/展示），**不参与权限判定**。权限判定统一由 `resource_permission` 表负责。创建资源时自动在 `resource_permission` 插入 WRITE 记录。

### 变更

所有资源实体统一使用 `created_by_user_id` 字段标识创建者：

| 表 | 当前字段 | 变更 |
|----|---------|------|
| problem | 无 | **新增** `created_by_user_id BIGINT NULL` |
| problem_set | `owner_user_id` | **重命名**为 `created_by_user_id` |
| class_info | `teacher_user_id` | **重命名**为 `created_by_user_id` |
| contest | 无（靠 contest_manager_rel.is_owner） | **新增** `created_by_user_id BIGINT NULL` |

## 四、统一权限服务：PermissionService

### 核心方法

```java
// 判断
boolean hasPermission(Long userId, ResourceType type, Long resourceId, PermissionOp op)
// 断言（失败抛异常）
void assertPermission(Long userId, ResourceType type, Long resourceId, PermissionOp op)
// 授权（调用者必须对该资源持有 WRITE）
void grantPermission(Long granterId, ResourceType type, Long resourceId, Long targetUserId, PermissionOp op)
// 撤销
void revokePermission(Long revokerId, ResourceType type, Long resourceId, Long targetUserId, PermissionOp op)
```

### 权限解析链（优先级从高到低）

```
1. ADMIN/SUPER_ADMIN 角色  → READ + WRITE 一切
2. 显式授权               → resource_permission 表中有匹配记录（WRITE 蕴含 READ）
                            注：创建者的 WRITE 也在此表中，创建时自动插入
3. 公开标记               → resource.is_public == true 且 op == READ → 通过
4. 继承访问               → 详见下方
5. RBAC 默认              → TEACHER 角色 → READ 一切 + 可创建新资源
6. 拒绝
```

> 注意：不额外进行 `created_by_user_id` 的权限判定。创建者的权限完全通过 `resource_permission` 表中的自动插入记录来保证。

### 继承访问规则（第 4 步）

| 资源类型 | 继承路径 | 条件 |
|---------|---------|------|
| PROBLEM_SET | 用户是班级成员 → 班级有**已发布的作业**关联此题单 | assignment.status = PUBLISHED |
| CONTEST | 用户是班级成员 → class_contest_rel 链接此竞赛 | 竞赛时间窗口校验保留（赛前/赛中/赛后） |
| PROBLEM | 通过所在 PROBLEM_SET 或 CONTEST 的权限间接获得 | 沿用当前 ProblemSourceType 三分支逻辑，但调用方改为 PermissionService |

### 竞赛时间窗口（特殊业务逻辑，纳入 PermissionService 内部）

竞赛的继承访问需叠加时间判断，逻辑与当前 ProblemAccessService 中一致：
- 赛前：仅 WRITE 权限者可访问
- 赛中：注册选手可访问
- 赛后：注册选手 + 班级成员 + 公开竞赛全员

### 替代现有代码

| 现有代码 | 处置 |
|---------|------|
| `ProblemAccessService` | **整体删除**，逻辑并入 PermissionService |
| `ProblemAccessMapper` + XML | **整体删除**，查询逻辑重写到 ResourcePermissionMapper |
| `ClassService.assertCanManageClass()` | 替换为 `permissionService.assertPermission(userId, CLASS, classId, WRITE)` |
| `ClassService.assertCanViewClassRelated()` | 替换为 `permissionService.assertPermission(userId, CLASS, classId, READ)` |
| `ClassService.assertTeacherOrAdmin()` | 删除 |
| `ProblemSetService.assertCanManageProblemSet()` | 替换为 `permissionService.assertPermission(userId, PROBLEM_SET, psId, WRITE)` |
| `ProblemSetService.assertCanViewProblemSet()` | 替换为 `permissionService.assertPermission(userId, PROBLEM_SET, psId, READ)` |
| `ProblemService.canCurrentUserViewPrivateProblems()` | 替换为 PermissionService 在列表查询时的过滤逻辑 |

## 五、中间资源：作业（Assignment）

### 解决的问题

教师把题单关联到班级后，学生立即可见。但教师可能想提前准备"未来作业"，暂不公开。

### Schema

```sql
CREATE TABLE assignment (
    id                   BIGINT       NOT NULL AUTO_INCREMENT,
    public_id            CHAR(36)     NOT NULL,
    class_id             BIGINT       NOT NULL,
    problem_set_id       BIGINT       NOT NULL,
    title                VARCHAR(255) NOT NULL,
    description          TEXT         NULL,
    status               VARCHAR(16)  NOT NULL DEFAULT 'DRAFT' COMMENT 'DRAFT|PUBLISHED|CLOSED',
    deadline             DATETIME     NULL     COMMENT 'NULL 表示无截止时间',
    created_by_user_id   BIGINT       NULL,
    created_at           TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at           TIMESTAMP    NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    is_del               TINYINT(1)   NOT NULL DEFAULT 0,
    PRIMARY KEY (id),
    UNIQUE INDEX uk_public_id (public_id),
    INDEX idx_class (class_id),
    INDEX idx_ps (problem_set_id),
    FOREIGN KEY (class_id) REFERENCES class_info(id),
    FOREIGN KEY (problem_set_id) REFERENCES problem_set(id),
    FOREIGN KEY (created_by_user_id) REFERENCES user_info(id)
);
```

### 生命周期

```
DRAFT ──(教师发布)──> PUBLISHED ──(截止/教师关闭)──> CLOSED
```

- **DRAFT**：仅对班级 WRITE 权限者可见（教师/管理员）
- **PUBLISHED**：班级全体成员可见，学生可通过作业访问关联的题单和题目
- **CLOSED**：学生仍可查看但不可提交（未来特性）

### 替代关系

`class_problem_set_rel` 表 → **删除**，由 `assignment` 替代。

## 九、Controller 注解简化

### 新约定

- `@RequireRole` **仅用于粗粒度端点准入**（如管理面板仅 ADMIN 可进入）
- 细粒度的"能否操作这个具体资源"**全部由 PermissionService 处理**，不在注解层做

### 具体变更

| Controller | 现状 | 改为 |
|-----------|------|------|
| ProblemController 创建/编辑/删除 | `@RequireRole({ADMIN, SUPER_ADMIN})` | 去掉注解（需登录即可），由 PermissionService 检查 WRITE 权限。TEACHER 可创建新题目（RBAC 默认），STUDENT 不可 |
| ClassController 创建 | `@RequireRole({TEACHER, ADMIN, SUPER_ADMIN})` | `@RequireRole({TEACHER, ADMIN, SUPER_ADMIN})` 不变 |
| ClassController 管理操作 | `@RequireRole({TEACHER, ADMIN, SUPER_ADMIN})` | 去掉注解，由 PermissionService 检查对具体班级的 WRITE |
| ProblemSetController 创建 | `@RequireRole({USER, ADMIN, SUPER_ADMIN})` | 去掉注解（需登录即可），PermissionService 的 RBAC 默认规则控制 |
| MeController heatmap | `@RequireRole({USER})` | `@RequireRole({STUDENT})` 或去掉注解（仅需登录） |
| SubmissionController 提交 | `@RequireRole({USER})` | `@RequireRole({STUDENT})` 或去掉注解 |
| CommonController 用户列表 | `@RequireRole({TEACHER, ADMIN, SUPER_ADMIN})` | 不变 |
| CommonController 批量导入 | `@RequireRole({ADMIN, SUPER_ADMIN})` | 不变 |

## 十、新增 API 端点

### 权限管理

```
POST   /api/permission/grant     — 授予权限（调用者需 WRITE）
DELETE /api/permission/revoke     — 撤销权限（调用者需 WRITE）
GET    /api/permission/{type}/{publicId} — 查看资源权限列表（调用者需 WRITE）
```

### 作业管理

```
POST   /api/class/{classPublicId}/assignment              — 创建作业（DRAFT）
GET    /api/class/{classPublicId}/assignment/page          — 作业列表（学生只看 PUBLISHED）
GET    /api/class/{classPublicId}/assignment/{publicId}    — 作业详情
PUT    /api/class/{classPublicId}/assignment/{publicId}    — 修改作业（含发布/关闭状态变更）
DELETE /api/class/{classPublicId}/assignment/{publicId}    — 删除作业
```

### 角色管理（管理员面板）

```
PUT    /api/admin/user/{userPublicId}/role   — 设置角色
```

权限矩阵：
- **ADMIN** 可设置目标用户为 STUDENT / TEACHER
- **SUPER_ADMIN** 可设置目标用户为 STUDENT / TEACHER / ADMIN
- SUPER_ADMIN 角色仅可通过数据库操作授予

### 需要删除的端点

```
PUT    /api/class/{id}/problem_set/{psId}     — 关联题单到班级（被作业替代）
```

## 十五、验证方案

### 权限解析正确性

对每种资源类型，覆盖以下场景组合：

| 场景 | 预期 |
|------|------|
| ADMIN/SUPER_ADMIN 访问私有资源 | READ+WRITE 通过 |
| 创建者访问自己的资源（通过 resource_permission 中的自动 WRITE 记录） | WRITE 通过 |
| 被显式授予 READ 的用户 | READ 通过，WRITE 拒绝 |
| 被显式授予 WRITE 的用户 | READ+WRITE 通过 |
| 公开资源 + 未授权学生 | READ 通过，WRITE 拒绝 |
| 私有资源 + 未授权学生 | READ+WRITE 拒绝 |
| 班级成员 + 已发布作业 | 可 READ 关联题单 |
| 班级成员 + DRAFT 作业 | 不可 READ 关联题单 |
| TEACHER 访问他人私有资源 | READ 通过（RBAC 默认），WRITE 拒绝 |
| WRITE 权限者授权他人 | 成功 |
| READ 权限者尝试授权他人 | 拒绝 |
| SUPER_ADMIN 授予 ADMIN 角色 | 成功 |
| ADMIN 尝试授予 ADMIN 角色 | 拒绝 |

### 端到端流程

1. 注册用户 → 确认角色为 STUDENT
2. ADMIN 登录 → 将用户提升为 TEACHER → 确认新角色
3. TEACHER 创建题目、题单、班级 → 确认自动拥有 WRITE
4. TEACHER 在班级内创建 DRAFT 作业 → STUDENT 看不到
5. TEACHER 发布作业 → STUDENT 可见作业、可访问关联题单和题目
6. TEACHER 分享题单给另一 TEACHER（授予 WRITE）→ 对方可编辑
7. STUDENT 尝试创建题目 → 拒绝（RBAC 默认无创建权限）
