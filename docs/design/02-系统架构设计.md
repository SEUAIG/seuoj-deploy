# SEUOJ 系统架构设计

## 1. 整体架构

```
                         ┌─────────────────────────────────┐
                         │         用户浏览器               │
                         └──────────────┬──────────────────┘
                                        │ HTTP :2280
                         ┌──────────────▼──────────────────┐
                         │        Nginx (frontend)          │
                         │  静态资源托管 + 反向代理           │
                         ├──────────────────────────────────┤
                         │  /           → SPA 静态文件       │
                         │  /api/*      → backend:8080      │
                         │  /agent/*    → agentend:8002     │
                         └──┬───────────────────┬───────────┘
                            │                   │
               ┌────────────▼──────┐  ┌────────▼──────────┐
               │   backend (Java)  │  │  agentend (Python) │
               │  Spring Boot 3.2  │  │    FastAPI + AI    │
               │     :8080         │  │      :8002         │
               └──┬──────────┬─────┘  └───────────────────┘
                  │          │
     ┌────────────▼──┐  ┌───▼──────────────┐
     │  MySQL 8.0    │  │  judgend (Rust)   │
     │    :3306      │  │  Axum + Seccomp   │
     │               │  │     :9090         │
     └───────────────┘  └──────────────────┘
```

## 2. 服务间通信

### 2.1 通信方式

所有服务间通信均为 **HTTP REST**，运行在 Docker 内部网络中：

| 调用方 | 被调方 | 协议 | 路径 | 说明 |
|--------|--------|------|------|------|
| 浏览器 | Nginx | HTTP | `:2280` | 唯一对外入口 |
| Nginx | backend | HTTP | `backend:8080/api/*` | API 反向代理 |
| Nginx | agentend | HTTP | `agentend:8002/agent/*` | Agent 反向代理 |
| backend | judgend | HTTP | `judgend:9090/judge/*` | 提交评测、题目管理 |
| judgend | backend | HTTP | `backend:8080/judge/*` | 评测结果回调 |
| backend | MySQL | TCP | `mysql:3306` | 数据库连接 |
| agentend | judgend | HTTP | `judgend:9090` | 代码执行（实验性） |

### 2.2 评测流程（关键链路）

```
用户提交代码
    │
    ▼
[frontend] POST /api/submission
    │
    ▼
[backend] SubmissionService.submit()
    ├─ 1. 写入 submission 记录 (status=PENDING)
    ├─ 2. 存储代码文件到本地磁盘
    └─ 3. POST /judge/submission → [judgend]
                                        │
                                        ▼
                               [judgend] 异步评测
                               ├─ 获取 semaphore 许可
                               ├─ 编译代码
                               ├─ 逐测试点运行（seccomp 沙箱）
                               ├─ 输出校验（standard/special/interactive）
                               ├─ 子任务评分聚合
                               └─ PUT /judge/submission/{id} → [backend]
                                                                   │
                                                                   ▼
                                                          [backend] JudgeService
                                                          └─ 更新 submission 记录
                                                             (status=FINISHED, verdict, score)
```

### 2.3 AI 问答流程

```
用户提问
    │
    ▼
[frontend] POST /agent/api/rag/messages/rag_answer
    │
    ▼
[agentend] SmartAnswer Pipeline
    ├─ 1. 推荐控制器：LLM 判断是否需要前置知识/题库推荐
    ├─ 2. FAISS 向量检索 + BM25 稀疏检索
    ├─ 3. PEV Agent 框架：
    │      ├─ Planner: 任务分解
    │      ├─ Executor: 检索 + LLM 合成答案
    │      └─ Verifier: 验证覆盖度与准确性（最多 3 轮迭代）
    ├─ 4. 生成引用（Citation）
    └─ 5. 持久化消息到 SQLite
```

## 4. 网络拓扑

### Docker Compose 网络

所有容器共享 Docker 默认网络（bridge），通过容器名互相发现：

```
Docker Network (bridge)
├── frontend  (nginx)     → 暴露 2280:80, 443:443
├── backend   (java)      → 内部 8080
├── judgend   (rust)      → 内部 9090
├── agentend  (python)    → 内部 8002
└── mysql     (mysql:8.0) → 内部 3306
```

**端口暴露策略**：仅 Nginx 对宿主机暴露端口，其余服务通过 Docker 内部网络通信，减少攻击面。

## 5. 安全架构

### 5.1 认证层

```
请求 → Nginx → backend
                 │
        ┌────────▼────────┐
        │ JwtAuthInterceptor│   提取 Bearer Token
        │  (Ed25519 验证)   │   设置 UserContext
        └────────┬────────┘
                 │
        ┌────────▼────────┐
        │   AuthAspect     │   @RequireRole / @AllowAnonymous
        │  (AOP 角色检查)   │   数据库查询角色
        └────────┬────────┘
                 │
        ┌────────▼────────┐
        │   Controller     │   业务逻辑
        └─────────────────┘
```

### 5.2 权限模型

系统采用两级权限模型：

1. **角色级权限（RBAC）**：`@RequireRole` 注解控制端点准入
2. **资源级权限**：`resource_permission` 表统一管理，支持 READ/WRITE 授权

权限解析链（优先级从高到低）：
1. ADMIN/SUPER_ADMIN 角色 → READ + WRITE 一切
2. 显式授权 → resource_permission 表中的匹配记录（WRITE 蕴含 READ）
3. 公开标记 → resource.is_public == true 且 op == READ → 通过
4. 继承访问 → 班级成员通过作业访问题目
5. RBAC 默认 → TEACHER 角色 → READ 一切 + 可创建新资源
6. 拒绝

### 5.3 判题端安全

```
用户代码 → [judgend]
              │
    ┌─────────▼──────────┐
    │   编译 (独立进程)     │
    └─────────┬──────────┘
              │
    ┌─────────▼──────────┐
    │   Seccomp 沙箱执行   │
    │  ├─ CPU 时间限制     │
    │  ├─ 内存限制         │
    │  ├─ 输出大小限制     │
    │  ├─ 进程数限制 (=1)  │
    │  └─ 系统调用白名单   │
    └─────────┬──────────┘
              │
    ┌─────────▼──────────┐
    │   Docker 容器隔离    │
    └────────────────────┘
```

### 5.4 服务间认证

| 链路 | 认证方式 |
|------|---------|
| 浏览器 → backend | JWT Bearer Token (Ed25519) |
| backend → judgend | Secret Key（环境变量 `JUDGE_SECRET`） |
| judgend → backend | 同一 Secret Key |

## 6. 扩展性分析

### 当前瓶颈

| 组件 | 瓶颈 | 影响 |
|------|------|------|
| judgend | Semaphore 并发数（默认 6） | 高峰期评测排队 |
| MySQL | 单实例，无读写分离 | 大量提交时数据库压力 |
| agentend | LLM API 调用延迟 | AI 功能响应慢 |
| Nginx | 单点，无负载均衡 | 无法水平扩展 |

### 可扩展方向

| 方向 | 方案 |
|------|------|
| 评测扩容 | judgend 多实例 + 消息队列分发 |
| 数据库扩容 | 读写分离 / 分库分表 |
| 前端 CDN | 静态资源上 CDN，Nginx 仅代理 API |
| 微服务化 | 拆分 backend 为独立服务（auth/problem/contest/class） |
| 缓存层 | Redis 缓存热点数据（题目详情、排行榜） |

## 7. 技术债与风险

| 风险项 | 说明 | 建议 |
|--------|------|------|
| 无消息队列 | backend → judgend 为同步 HTTP 调用 | 引入 RabbitMQ/Redis Stream |
| 无监控告警 | 缺少 Prometheus/Grafana 集成 | 添加指标采集与告警 |
| 无 CI/CD | 手动构建部署 | 配置 GitHub Actions |
| 会话存储分散 | agentend 用 SQLite，backend 用 MySQL | 统一存储方案 |
| 无限流 | API 无 rate limiting | 添加 Nginx 限流或 Spring Cloud Gateway |
| 日志分散 | 各服务独立日志文件 | 集中式日志（ELK/Loki） |
