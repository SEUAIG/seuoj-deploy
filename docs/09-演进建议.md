## 2. 高优先级改进（P0 - 稳定性与可用性）

### 2.1 添加 CI/CD 流水线

**现状**: 手动构建部署，无自动化测试。

**建议**:
- GitHub Actions：PR 触发 → 构建 → 单元测试 → 集成测试 → 部署
- 各子模块独立 CI：backend (mvn test), frontend (npm run lint + build), judgend (cargo test)
- 主仓库 CD：合并到 main 后自动部署到测试环境

### 2.2 引入消息队列

**现状**: backend → judgend 为同步 HTTP 调用，提交高峰时可能超时或丢失。

**建议**:
- 引入 Redis Stream 或 RabbitMQ
- 提交流程：backend → 入队 → judgend 消费
- 好处：削峰填谷、断点续传、评测进度通知

### 2.3 添加监控告警

**现状**: 无监控，依靠查看容器日志排错。

**建议**:
- Prometheus + Grafana：采集 JVM、MySQL、API 延迟等指标
- ELK 或 Loki：集中式日志
- 告警规则：评测延迟 > 30s、MySQL 连接池耗尽、5xx 错误率 > 1%
- 各服务添加 `/health` 端点，Docker healthcheck 配置

### 2.4 HTTPS 配置

**现状**: 443 端口已暴露但未配置证书。

**建议**:
- Nginx 配置 Let's Encrypt 自动续签
- HTTP 强制跳转 HTTPS
- 添加安全头（HSTS, CSP, X-Frame-Options）

## 5. 架构演进路线图

### 阶段一：稳定化（1-2 个月）

```
当前状态                          目标状态
────────                          ────────
无 CI/CD          ──────────►    GitHub Actions CI
无监控            ──────────►    Prometheus + Grafana 基础监控
HTTP only         ──────────►    HTTPS + 安全头
同步评测调用      ──────────►    Redis Stream 异步队列
无限流            ──────────►    Nginx 限流 + 后端限流
```

### 阶段二：功能完善（2-4 个月）

```
基础竞赛          ──────────►    完整竞赛系统 + Rating
基础班级          ──────────►    教学分析 + 成绩导出
空讨论区          ──────────►    题目讨论 + 评论
AI 原型           ──────────►    接入真实数据 + 错题诊断
```

### 阶段三：规模化（4-6 个月）

```
单实例部署        ──────────►    Kubernetes 编排
单数据库          ──────────►    读写分离 + Redis 缓存
单评测节点        ──────────►    多节点评测集群
集中式存储        ──────────►    对象存储 (MinIO/S3)
```

## 6. 技术债清单

| # | 类型 | 描述 | 影响 | 建议 |
|---|------|------|------|------|
| 1 | 安全 | Agent 服务无鉴权 | 任意用户可伪造身份 | 透传 JWT 或添加内部认证 |
| 2 | 数据 | 题面存 judgend，元数据存 MySQL | 数据分散，一致性风险 | 统一到 MySQL 或引入版本控制 |
| 3 | 数据 | Agent 用 SQLite，backend 用 MySQL | 会话数据孤立 | 统一到 MySQL |
| 4 | 配置 | 开发模式固定验证码 | 生产环境安全隐患 | 环境区分配置 |
| 5 | 测试 | 缺少端到端测试 | 回归风险 | 添加 Playwright E2E 测试 |
| 6 | 代码 | Agent code_generator.py 大部分注释 | 死代码 | 清理或完成实现 |
| 7 | 依赖 | 前端代理指向硬编码 IP | 换环境需改配置 | 统一使用环境变量 |
| 8 | 文档 | 各子模块 API 文档缺失 | 协作困难 | Swagger/OpenAPI 文档 |
| 9 | 运维 | Docker healthcheck 未配置 | 容器异常无法自动恢复 | 添加 health check |
| 10 | 运维 | 无 Docker Secrets | 密钥明文存储 | 使用 Docker Secrets / Vault |

## 7. 建议优先行动项

> 按照投入产出比排序，前 5 项建议立即着手：

1. **配置 GitHub Actions CI** — 各子模块构建+测试自动化，防止回归
2. **引入 Redis** — 用作评测队列 + 热点缓存，一举两得
3. **HTTPS + 安全头** — 基本安全要求，配置简单
4. **Agent JWT 鉴权** — 堵住身份伪造漏洞
5. **Docker healthcheck** — 自动恢复挂掉的容器，提升可用性
