## 3. 评测流程

```
1. 接收请求
   POST /judge/submission
   Body: { submissionId, pid, code, language }
   → 立即返回 { code: 0, message: "Success" }（异步处理）

2. 并发控制
   获取 Semaphore 许可（默认最大 6 并发）
   → Tokio spawn 异步任务

3. 编译阶段
   ├─ 写代码到临时文件（按语言添加扩展名）
   ├─ C/C++: gcc/g++ 编译为可执行文件
   │         C++11/17/20 添加 -std 标志
   ├─ Java: javac 编译，java 执行
   ├─ Python/Node: 直接解释执行
   ├─ Go: go build 构建
   └─ 编译失败 → CompileError，回调 backend

4. 加载配置
   读取题目目录下的 info.toml
   解析：problem_type, checker_type, time/memory limits
   拓扑排序子任务（处理依赖关系）

5. 逐测试点执行
   For each subtask (拓扑序):
     For each testcase:
       ├─ 配置 judger::Config:
       │   exe_path, input/output/error/log paths
       │   time limit, memory limit, seccomp rule
       ├─ 调用 judger::run() — Seccomp 沙箱内执行
       ├─ 解析结果: Success/TLE/MLE/RE/SE
       │
       ├─ Standard: 逐行文本比较（忽略尾部空白和空行）
       ├─ Special:  运行 checker 程序
       │   命令: ./checker <input> <output> <answer>
       │   exit 0 = AC, exit 7 = Partial (解析 stderr 中 "points X")
       └─ Interactive: 交互器与用户程序通过 pipe 通信

6. 评分聚合
   ├─ 每个测试点计算得分
   ├─ 子任务评分策略: min (取最小) / sum (求和)
   └─ 截断输出到 AIJ_OUTPUT_TRUNCATE_LENGTH (默认 200 字符)

7. 回调 backend
   PUT {backend_host}:{backend_port}{prefix}/judge/submission/{submissionId}
   Body: { status, resultDetail (JSON), subtasks, score }

8. 清理
   如果 AIJ_SAVE_SUBMISSIONS=false，删除提交临时目录
```

## 4. 支持语言

| 语言 | 编译器/解释器 | 编译标志 | 时间倍率 | 内存倍率 | Seccomp 规则 |
|------|-------------|---------|---------|---------|-------------|
| C | gcc | 默认 | 1x | 1x | CCpp |
| C++ | g++ | 默认 | 1x | 1x | CCpp |
| C++11 | g++ | `-std=c++11` | 1x | 1x | CCpp |
| C++17 | g++ | `-std=c++17` | 1x | 1x | CCpp |
| C++20 | g++ | `-std=c++20` | 1x | 1x | CCpp |
| Python 3.12 | python3 | 直接执行 | 2x | 1x | Python |
| Java 17 | javac + java | `-cp`, `-Xmx` | 2x | 2x | Java |
| Node.js | node | 直接执行 | 2x | 1x | Node |
| Go 1.22 | go build | 构建 | 1x | 2x | Golang |

## 6. 沙箱安全机制

### 资源限制

| 维度 | 参数 | 说明 |
|------|------|------|
| CPU 时间 | `max_cpu_time` | 毫秒，题目配置 |
| 实际时间 | `max_real_time` | = 2 × CPU 时间 |
| 内存 | `max_memory` | 字节，从 `memory_limit_kb * 1024` |
| 栈空间 | `max_stack` | 同内存限制 |
| 输出大小 | `max_output_size` | 防止磁盘耗尽 |
| 进程数 | `max_process_number` | 固定为 1 |

### Seccomp 规则

每种语言有独立的系统调用白名单：
- **CCpp**: 最严格，仅允许基本 I/O 系统调用
- **Python**: 允许 Python 运行时所需的额外系统调用
- **Java**: 允许 JVM 进程管理相关系统调用
- **Node**: 允许 Node.js 运行时系统调用
- **Golang**: 允许 Go 运行时系统调用

### 容器隔离

Docker 容器 (Ubuntu 24.04) 提供第一层隔离，Seccomp 提供第二层内核级隔离。

## 7. 题目配置格式

### problem.json（题目元数据）

```json
{
  "pid": "test01",
  "title": "题目标题",
  "description": "题目描述（Markdown）",
  "input_format": "输入格式说明",
  "output_format": "输出格式说明",
  "examples": [
    { "input": "样例输入", "output": "样例输出" }
  ]
}
```

### info.toml（评测配置）

```toml
[problem_info]
problem_type = "Standard"           # Standard | Interactive | Special
checker_type = "Standard"           # Standard | Special | Interactor
time_limit_ms = 1000
memory_limit_kb = 262144

[[testcases]]
input = "data/1.in"
answer = "data/1.ans"
# 可选：time_limit_ms, memory_limit_kb (覆盖全局)

[[subtasks]]
id = 1
score = 100
testcases = [1]
strategy = "min"                    # min | sum
depends_on = []                     # 子任务依赖

[custom_modules]
checker = "data/checker.cpp"        # Special Judge 源码
interactor = "data/interactor.cpp"  # 交互器源码
```

### 题目目录结构

```
assets/problems/{pid}/
├── problem.json          # 题目元数据
├── info.toml             # 评测配置
└── data/
    ├── 1.in              # 测试输入
    ├── 1.ans             # 期望输出
    ├── checker.cpp       # Special Judge 源码 (可选)
    ├── checker           # 编译后的 checker (可选)
    ├── interactor.cpp    # 交互器源码 (可选)
    └── interactor        # 编译后的交互器 (可选)
```

## 8. 评测结果数据结构

### JudgeResult（回调 backend 的数据）

```
CompileError:
  { status: "CompileError", detail: "错误信息" }

MaybeError:
  {
    status: "Success",
    resultDetail: [              # 每个测试点
      {
        case_id: 1,
        verdict: "Accepted",     # AC/WA/TLE/MLE/RE/SE
        time_ms: 50,
        memory_kb: 1024,
        score: 100,
        input: "...",            # 截断后的输入
        output: "...",           # 截断后的用户输出
        answer: "...",           # 截断后的期望输出
        error: "..."             # stderr 输出
      }
    ],
    subtasks: [                  # 子任务评分
      { id: 1, score: 100, max_score: 100 }
    ],
    score: 100                   # 总分
  }
```

## 10. Dockerfile

两阶段构建：

**阶段 1 (builder)**: `rust:latest`
- 安装 `libseccomp-dev`, `pkg-config`
- `cargo build --release`

**阶段 2 (runtime)**: `ubuntu:24.04`
- 安装运行时依赖：libseccomp2, GCC 13, OpenJDK 21, Go 1.22, Python 3.12, Node.js 18+
- 复制编译产物
- 入口：`./runner`

## 12. 并发模型

- **Tokio 异步运行时**：完整特性启用
- **Semaphore 并发控制**：限制最大同时评测任务数
- **异步 Spawn**：每个提交在独立 Task 中评测，不阻塞 HTTP 响应
- **线程安全配置**：`RwLock<HashMap>` 缓存二进制路径
- **非阻塞 I/O**：文件操作和网络调用均为异步
